# 测试用例生成器

根据输入源自动生成规范、全面的测试用例。

---

## 一、用例编写总体原则（必须遵守）

| 原则 | 说明 | 示例 |
|------|------|------|
| **唯一性** | 一个用例只验证一个核心场景/一个点 | ❌ 一个用例同时测登录+注册<br>✅ 一个用例只测"密码为空"场景 |
| **可复现** | 步骤清晰、环境明确、无随机不可控因素 | ❌ "随机输入测试"<br>✅ "输入固定测试数据：13800138000" |
| **完整性** | 覆盖正常、异常、边界、空值、错误格式 | 必须覆盖5大类场景 |
| **可断言** | 所有结果必须可量化、可校验，不能模糊 | ❌ "看看是否正常"<br>✅ "返回状态success，日期从2026-01-01开始" |
| **规范性** | 字段统一、命名统一、格式统一 | 用例ID统一格式：模块_功能_序号 |

---

## 二、测试用例标准字段

### 1. 用例 ID（必填）

**定义**：用例唯一标识，不重复

**命名规则**：`模块_功能_序号`

**示例**：
- `Login_Account_001`（登录模块-账号相关-第1个用例）
- `Order_Create_003`（订单模块-创建订单-第3个用例）
- `Weather_Query_012`（天气查询模块-查询功能-第12个用例）

---

### 2. 用例名称（必填）

**定义**：清晰说明"测什么、什么场景、预期结果"

**格式**：`模块 + 场景 + 预期`

**示例**：
- `日期生成 - 正常区间 - 按天递增`
- `退款工具 - 订单不存在 - 返回失败`
- `天气查询 - 城市为空 - 参数校验失败`
- `用户登录 - 密码为空 - 提示必填`

---

### 3. 所属模块（必填）

**定义**：归属的功能模块

**示例**：
- 实况一览
- 资产管理
- 因子管理
- 登录模块
- 订单管理
- 用户中心

---

### 4. 优先级（必填）

| 级别 | 定义 | 适用场景 | 执行要求 |
|------|------|---------|---------|
| **P0** | 阻塞级：核心主流程，直接影响系统运行，不修复无法继续测试 | 登录功能崩溃、支付功能失败、系统无法启动 | 立即执行，发现问题立即反馈，优先修复 |
| **P1** | 高级：重要功能，影响用户核心场景，不修复无法完成关键操作，但不阻塞其他功能 | 密码修改失败、订单提交后无法查看、商品查询无结果 | P0完成后优先执行 |
| **P2** | 中级：次要功能、优化类，不影响核心使用，仅影响体验或边缘场景 | 提示文案不规范、页面样式轻微错乱、按钮点击无反馈（功能正常） | P0、P1完成后执行 |
| **P3** | 低级：极边缘场景、异常场景（几乎不发生），不影响用户使用 | 连续点击100次无异常、输入特殊字符后页面无错乱 | 最后执行，可根据时间灵活调整 |

---

### 5. 前置条件（必填）

**定义**：执行用例前必须满足的环境/状态

**示例**：
- 技能已加载
- 配置参数正确
- 依赖服务可调用（或已Mock）
- 测试数据已准备
- 用户已登录
- 订单已创建

---

### 6. 输入数据（必填）

**定义**：用户输入、工具入参、请求参数、环境变量

**必须包含**：
- 用户指令
- 入参字段
- 关键数据

**示例**：
```
用户输入：查询 2026-01-01 到 2026-03-26 的日期
参数：{startDate: "2026-01-01", endDate: "2026-03-26"}
```

```
用户输入：登录账号
参数：{username: "13800138000", password: "Test@123"}
```

---

### 7. 测试步骤（必填）

**定义**：一步步怎么操作，必须清晰可复现

**格式**：编号列表，每步一个动作

**示例**：
```
步骤1：传入开始日期 2026-01-01 和结束日期 2026-03-26
步骤2：执行日期生成工具
步骤3：获取返回结果
步骤4：校验日期是否连续
```

```
步骤1：打开登录页面
步骤2：输入用户名 13800138000
步骤3：输入密码 Test@123
步骤4：点击登录按钮
步骤5：观察页面跳转和用户信息显示
```

---

### 8. 预期结果（必填，最重要）

**定义**：用例执行完成后，必须出现的明确结果

**要求**：可观察、可验证、可量化

**示例**：
```
- 返回状态 success
- 日期从 2026-01-01 开始
- 每天 +1，无跳天
- 最后一天为 2026-03-26
- 无报错、无异常
```

```
- 登录成功，跳转至首页
- 右上角显示用户名 "13800138000"
- Token 有效期 24小时
- 无报错、无异常
```

**❌ 错误示例**：
- "看看是否正常"（模糊，不可验证）
- "功能正常"（无具体判断标准）
- "应该能登录"（不确定）

---

### 9. 备注（可选）

**内容**：
- 备注说明
- 特殊说明
- 依赖说明
- 关联Bug/需求
- 数据来源说明

---

## 三、用例必须覆盖的场景（5大类）

### 1. 正常场景

**定义**：正确输入 → 正确输出

**覆盖要点**：
- 主流程、最常用逻辑
- 核心功能正常执行
- 标准数据格式

**示例**：
- 登录-账号密码正确-登录成功
- 查询-城市有效-返回天气数据
- 订单-商品存在-创建成功

---

### 2. 边界场景

**定义**：极限值、临界条件

**覆盖要点**：
- 开始日期 = 结束日期
- 最大长度、最小长度
- 闰年、月末、跨月
- 数值范围边界（最小值、最大值）
- 字段长度边界（空、满）

**示例**：
- 密码长度6位（最小边界）- 登录成功
- 密码长度20位（最大边界）- 登录成功
- 开始日期等于结束日期 - 返回单日数据
- 订单金额0.01元（最小）- 创建成功

---

### 3. 异常场景

**定义**：非预期输入、错误数据

**覆盖要点**：
- 入参为空（必填字段缺失）
- 入参格式错误
- 开始日期 > 结束日期
- 无效参数、非法字符
- 数据不存在

**示例**：
- 用户名为空 - 提示"用户名必填"
- 密码格式错误（含特殊字符）- 提示"密码格式不正确"
- 查询城市不存在 - 提示"城市名称无效"
- 日期格式错误（2026/01/01而非2026-01-01）- 提示"日期格式不正确"

---

### 4. 错误处理

**定义**：系统层面的异常处理

**覆盖要点**：
- 服务超时
- 无权限
- 资源不存在
- 网络异常
- 依赖服务不可用

**示例**：
- 登录服务超时 - 提示"系统繁忙，请稍后重试"
- 无权限访问订单 - 提示"权限不足"
- 查询服务不可用 - 返回错误码503
- 网络断开 - 本地缓存生效或提示重试

---

### 5. 兼容性/格式

**定义**：不同环境、格式下的表现

**覆盖要点**：
- 日期格式（YYYY-MM-DD、YYYY/MM/DD等）
- 编码格式（UTF-8、GBK）
- 字段类型（字符串、数字、布尔）
- 返回结构（JSON、XML）
- 浏览器兼容（Chrome、Firefox、Edge）
- 移动端/PC端适配

**示例**：
- 日期格式YYYY/MM/DD - 自动转换或提示格式错误
- 中文城市名 - 正常查询
- 返回JSON格式 - 字段完整、类型正确
- Chrome浏览器 - 功能正常显示

---

## 四、禁止编写的用例

| 禁止类型 | 说明 | ❌ 错误示例 |
|---------|------|-----------|
| 无明确预期结果 | 预期模糊、无法验证 | "看看是否正常"、"功能正常" |
| 一个用例测多个功能 | 违反唯一性原则 | "测试登录和注册功能" |
| 不可复现 | 依赖随机数据、环境不确定 | "随机输入测试"、"找个用户测试" |
| 无断言 | 无法自动校验结果 | "手动检查结果"、"看一眼就行" |
| 步骤模糊不清 | 操作不明确、无法执行 | "随便输入点什么"、"正常操作" |

## 五、输入类型处理

### 1. 需求文档（文件）

- 读取文档内容（支持 .docx, .doc, .md, .txt, .pdf）
- 提取功能点、业务规则、数据约束
- 分析用户场景和业务流程
- 识别隐含的测试需求

### 2. 禅道需求链接

- 调用禅道 API 获取需求详情
- 解析需求描述、验收标准
- 关联已有 Bug 和任务信息
- 输出时可导入禅道

### 3. 功能描述（文字）

- 用户直接描述功能
- 分析输入、输出、处理逻辑
- 梳理前置条件和依赖关系
- 提取关键字段和业务规则

### 4. API 接口文档

- 解析接口定义（URL、Method、参数）
- 分析请求参数（必填/可选、类型、范围）
- 分析响应结构（成功/失败场景）
- 生成 API 专项测试用例

## 六、输出格式

### Excel 输出（默认）

生成 `.xlsx` 文件，包含：

**Sheet1：测试用例列表**
| 用例ID | 用例名称 | 所属模块 | 优先级 | 前置条件 | 输入数据 | 测试步骤 | 预期结果 | 备注 |

- 按模块统计用例数量
- 按优先级统计用例数量
- 按测试类型统计覆盖率

**Sheet3：测试覆盖分析**
- 需求覆盖情况
- 遗漏场景提示
- 风险提示

生成 `.docx` 文件，结构：

```
1. 测试范围概述

3. 测试用例明细（表格形式）
4. 测试覆盖分析
5. 风险提示
```

### 禅道导入格式

生成符合禅道导入要求的 CSV 文件，提供导入步骤说明。

## 七、生成流程

### Step 1: 分析输入

- 确认输入类型（文档/禅道/描述/API）
- 提取功能点、业务规则、数据约束
- 列出所有可测试场景

### Step 2: 设计测试场景

按5大类维度拆分：

1. **正常场景**：正确输入 → 正确输出
2. **边界场景**：极限值、临界条件
3. **异常场景**：非预期输入、错误数据
4. **错误处理**：系统异常处理
5. **兼容性/格式**：不同环境、格式

### Step 3: 编写用例

每个测试场景生成独立用例：

- 用例ID按模块_功能_序号编排
- 用例名称格式：模块 + 场景 + 预期
- 步骤清晰编号、可复现
- 预期结果明确、可量化
- 输入数据具体、真实

### Step 4: 覆盖检查

自检是否遗漏：

- 是否覆盖所有功能点？
- 是否覆盖所有输入字段？
- 是否覆盖所有业务规则？
- 是否覆盖5大类场景？
- 是否有违反禁止规则？

### Step 5: 输出文件

- 默认生成 Excel 文件
- 用户指定 Word 则生成 Word
- 有禅道链接则提供导入方案
