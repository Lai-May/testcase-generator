## 一、用例编写总体原则（必须遵守）

> ⚠️ **这是测试用例编写的基本准则，所有用例必须遵守！**

### 📋 五大核心原则

| 原则 | 定义 | ✅ 正确示例 | ❌ 错误示例 |
|------|------|-----------|-----------|
| **唯一性** | 一个用例只验证一个核心场景/一个点 | 用例只测"密码为空"这一场景 | 一个用例同时测登录+注册 |
| **可复现** | 步骤清晰、环境明确、无随机不可控因素 | 输入固定数据：13800138000 | "随机输入测试"、"随便测试" |
| **完整性** | 覆盖正常、异常、边界、空值、错误格式 | 必须覆盖5大类全部场景 | 只写正常场景，忽略异常 |
| **可断言** | 所有结果必须可量化、可校验，不能模糊 | 返回状态success，显示用户名 | "看看是否正常"、"功能正常" |
| **规范性** | 字段统一、命名统一、格式统一 | 用例ID：Login_Account_001 | 用例ID随意命名：test1、case_a |

### 💡 原则检查清单

编写每条用例前，先检查：

```
□ 唯一性检查：这条用例只测一个场景吗？
□ 可复现检查：别人能按步骤执行吗？
□ 完整性检查：5大类场景都覆盖了吗？
□ 可断言检查：预期结果能被验证吗？
□ 规范性检查：字段格式统一吗？
```

## 二、测试用例标准字段

> 📝 每条测试用例必须包含以下字段，确保信息完整、规范。

### 字段一览表

| 序号 | 字段名称 | 必填 | 说明 |
|:----:|---------|:----:|------|
| 1 | 用例 ID | ✅ | 唯一标识，格式：`模块_功能_序号` |
| 2 | 用例名称 | ✅ | 格式：`功能 + 场景 + 预期` |
| 3 | 所属模块 | ✅ | 归属的功能模块名称 |
| 4 | 优先级 | ✅ | P0/P1/P2/P3 |
| 5 | 前置条件 | ✅ | 执行前必须满足的环境/状态 |
| 6 | 输入数据 | ✅ | 具体的测试数据、参数值 |
| 7 | 测试步骤 | ✅ | 清晰编号的操作步骤 |
| 8 | 预期结果 | ✅ | 可观察、可验证、可量化的结果 |
| 9 | 备注 | ❌ | 补充说明、特殊说明 |

### 1️⃣ 用例 ID（必填）

**命名规则**：`模块_功能_序号`

**命名示例**：

| 模块 | 功能 | 序号 | 完整 ID |
|------|------|------|---------|
| 登录 | 账号 | 001 | `Login_Account_001` |
| 订单 | 创建 | 003 | `Order_Create_003` |
| 天气 | 查询 | 012 | `Weather_Query_012` |
| 用户 | 注册 | 005 | `User_Register_005` |
| 支付 | 微信 | 002 | `Payment_Wechat_002` |

**命名规范**：
- 模块名使用英文或拼音首字母大写
- 功能名使用英文或拼音首字母大写
- 序号使用3位数字（001-999）
- 用下划线 `_` 连接各部分

---

### 2️⃣ 用例名称（必填）

**命名格式**：`功能 + 场景 + 预期结果`

**优秀命名示例**：

| ✅ 优秀命名 | 分析 |
|-----------|------|
| `登录 - 密码为空 - 提示必填` | 功能明确、场景具体、预期清晰 |
| `订单创建 - 商品不存在 - 返回失败` | 三段式结构完整 |
| `天气查询 - 城市为空 - 参数校验失败` | 可直接理解测试意图 |
| `支付 - 余额不足 - 提示充值` | 简洁但不失关键信息 |

**❌ 错误命名示例**：

| ❌ 错误命名 | 问题 |
|-----------|------|
| `测试登录` | 太笼统，不知道测什么 |
| `密码测试` | 缺少场景和预期 |
| `功能正常` | 缺少具体功能名称 |
| `登录功能测试用例1` | 冗余，未体现测试场景 |

### 4️⃣ 优先级（必填）

> 优先级决定了用例执行顺序和问题严重程度评估。

#### 优先级定义表

| 级别 | 名称 | 定义 | 典型场景 | 执行时机 |
|:----:|------|------|---------|---------|
| **P0** | 🔴 阻塞级 | 核心主流程，直接影响系统运行，不修复无法继续测试 | 登录崩溃、支付失败、系统无法启动 | 立即执行，发现问题立即反馈 |
| **P1** | 🟠 高级 | 重要功能，影响用户核心场景，不修复无法完成关键操作 | 密码修改失败、订单无法查看 | P0完成后优先执行 |
| **P2** | 🔵 中级 | 次要功能，仅影响体验或边缘场景 | 提示文案不规范、样式轻微错乱 | P0/P1完成后执行 |
| **P3** | ⚪ 低级 | 极边缘场景，几乎不发生 | 连续点击100次无异常 | 最后执行，时间灵活 |

#### 优先级判断流程图

```
                功能是否为核心主流程？
                       │
              ┌────────┴────────┐
              │                 │
             YES               NO
              │                 │
              ▼                 ▼
           P0 阻塞级     功能是否影响用户核心场景？
                              │
                     ┌────────┴────────┐
                     │                 │
                    YES               NO
                     │                 │
                     ▼                 ▼
                  P1 高级      仅影响体验或边缘场景？
                                     │
                            ┌────────┴────────┐
                            │                 │
                           YES               NO
                            │                 │
                            ▼                 ▼
                         P2 中级           P3 低级
```

#### P0 场景识别要点

**必须标记为 P0 的场景**：

| 类型 | 场景 | 原因 |
|------|------|------|
| 🚪 入口功能 | 登录、注册、首页加载 | 无法登录则所有功能都无法测试 |
| 💰 核心交易 | 支付、订单提交、退款 | 直接影响业务收入 |
| 🔐 安全相关 | 权限控制、数据安全 | 安全漏洞可能导致严重后果 |
| 📊 关键数据 | 数据保存、数据同步 | 数据丢失不可接受 |

### 5️⃣ 前置条件（必填）

**定义**：执行用例前必须满足的环境、状态、数据准备

**前置条件类型表**：

| 类型 | 示例 |
|------|------|
| **账号状态** | 用户已登录、账号已注册、账号已锁定、账号已注销 |
| **数据准备** | 测试数据已准备、订单已创建、商品已上架 |
| **权限状态** | 有新增权限、有删除权限、无编辑权限 |
| **服务状态** | 依赖服务可调用、数据库已连接、缓存已清空 |
| **环境配置** | 配置参数正确、环境变量已设置、网络正常 |

**⚠️ 注意**：
- 前置条件必须可验证、可达成
- 不可写"随机找一个用户"这种不可控条件
- 如果前置条件复杂，需在备注中说明如何准备

### 6️⃣ 输入数据（必填）

**定义**：具体的测试数据，包括用户输入、接口参数、环境变量等

**输入数据要素**：

| 要素 | 说明 | 示例 |
|------|------|------|
| 用户指令 | 用户说了什么 | "查询北京天气" |
| 参数字段 | 接口/工具参数 | `{city: "北京", date: "2026-04-01"}` |
| 关键数据 | 具体数值 | 手机号：13800138000，密码：Test@123 |

**输入数据格式规范**：

```
格式一：表格形式
┌─────────────────────────────────────┐
│ 字段名        │ 值                  │
├─────────────────────────────────────┤
│ username      │ 13800138000         │
│ password      │ Test@123            │
│ captcha       │ 1234                │
└─────────────────────────────────────┘

格式二：JSON 形式
{
  "username": "13800138000",
  "password": "Test@123",
  "captcha": "1234"
}

格式三：文字描述
用户输入手机号：13800138000
用户输入密码：Test@123
用户输入验证码：1234
```

### 7️⃣ 测试步骤（必填）

**定义**：一步步的操作指南，任何人都能按步骤执行

**步骤编写规范**：

| 规范 | 说明 |
|------|------|
| ✅ 编号格式 | 使用"步骤1、步骤2"或"1. 2. 3." |
| ✅ 每步一个动作 | 不要在一个步骤里写多个操作 |
| ✅ 动作明确 | 使用"点击"、"输入"、"选择"、"观察"等动词 |
| ✅ 数据具体 | 步骤中包含具体数据，不要写"输入某个值" |
| ❌ 避免模糊 | 不要写"正常操作"、"随便输入" |

**优秀步骤示例**：

```
步骤1：打开登录页面 https://example.com/login
步骤2：在用户名输入框输入 13800138000
步骤3：在密码输入框输入 Test@123
步骤4：点击"登录"按钮
步骤5：观察页面跳转情况
步骤6：检查右上角是否显示用户名
```

**❌ 错误步骤示例**：

```
❌ 步骤1：打开页面正常登录
❌ 步骤2：输入用户名和密码
❌ 步骤3：点击登录看看结果
```

### 8️⃣ 预期结果（必填，最重要）

> ⭐ **这是判断测试是否通过的核心依据，必须明确、可验证！**

**预期结果三要素**：

| 要素 | 说明 | 示例 |
|------|------|------|
| **可观察** | 能看到的现象 | 页面跳转、弹窗显示、数据变化 |
| **可验证** | 能对比的标准 | 状态码、返回值、字段值 |
| **可量化** | 能计数的指标 | 数量、时间、百分比 |

**优秀预期结果示例**：

```
✅ 示例1：登录成功场景
- 页面跳转至首页（URL 变为 https://example.com/home）
- 右上角显示用户名"13800138000"
- 返回 Token 有效期 24小时
- 无报错、无异常弹窗

✅ 示例2：密码为空场景
- 未发起登录请求（网络面板无请求）
- 密码输入框下方显示红色提示"密码不能为空"
- 登录按钮保持禁用状态
- 页面停留在登录页

✅ 示例3：订单创建成功
- 返回状态码 200
- 返回订单ID：ORD20260401001
- 订单状态为"待支付"
- 商品库存减少1
```

**❌ 错误预期结果示例**：

```
❌ "看看是否正常"        → 不可验证
❌ "功能正常"            → 无具体判断标准
❌ "应该能登录"          → 不确定
❌ "页面显示正确"        → 什么是"正确"？
❌ "用户体验良好"        → 主观判断，不可量化
```

## 三、用例必须覆盖的场景（5大类）

> 🎯 **每条功能必须覆盖以下5大类场景，确保测试完整！**

### 📊 场景覆盖率要求

| 场景类型 | 建议占比 | 最低条数 | 说明 |
|---------|:-------:|:-------:|------|

| 边界场景 | 15% | 2条 | 每个边界值至少1条 |
| 异常场景 | 25% | 3条 | 必填校验至少各1条 |
| 错误处理 | 15% | 2条 | 权限、网络异常 |
| 兼容性 | 15% | 2条 | 浏览器、格式兼容 |

### 1️⃣ 正常场景 ✅

**定义**：正确输入 → 正确输出，验证核心功能

#### 正常场景覆盖要点

| 要点 | 说明 | 示例用例 |
|------|------|---------|
| 主流程验证 | 核心功能正常执行 | 登录成功、订单创建成功 |
| 标准数据 | 使用标准格式的数据 | 11位手机号、6-20位密码 |
| 完整参数 | 所有必填字段正确填写 | 全部字段正确填写保存成功 |
| 常用组合 | 最常用的参数组合 | 微信支付+正常金额 |

#### 正常场景模板

```
用例名称：<功能> - 正常输入 - <预期结果>
输入数据：使用标准格式的有效数据
预期结果：功能正常执行，返回正确结果
```

---

### 2️⃣ 边界场景 🔶

**定义**：极限值、临界条件，验证系统边界处理能力

#### 边界值测试矩阵

| 边界类型 | 测试点 | 示例 |
|---------|-------|------|
| **长度边界** | 最小值、最大值、最小-1、最大+1 | 密码6位/20位/5位/21位 |
| **数值边界** | 最小值、最大值、零值、负值 | 金额0.01/99999/0/-1 |
| **时间边界** | 当天、昨天、明天、闰年、月末 | 2026-02-28（平年）/2024-02-29（闰年） |
| **数量边界** | 0个、1个、N个、N+1个 | 空列表/单条数据/最大数量 |
| **范围边界** | 起点值、终点值、范围外 | 开始日期=结束日期/开始>结束 |

#### 边界值公式

```
边界测试点 = { 最小值, 最小值-1, 最小值+1, 最大值, 最大值-1, 最大值+1 }

示例：密码长度6-20位
测试点 = { 6位(最小), 5位(最小-1), 7位(最小+1), 20位(最大), 19位(最大-1), 21位(最大+1) }
```

#### 边界场景模板

```
用例名称：<字段> - <边界类型> - <预期结果>
输入数据：<边界值的具体数据>
预期结果：<边界值处理结果>
备注：<边界值说明>
```

### 3️⃣ 异常场景 ❌

**定义**：非预期输入、错误数据，验证系统容错能力

#### 异常场景分类表

| 异常类型 | 测试点 | 示例 |
|---------|-------|------|
| **空值异常** | 必填字段为空 | 用户名空、密码空、参数空 |
| **格式异常** | 格式不符合要求 | 手机号非11位、邮箱无@、日期格式错误 |
| **类型异常** | 类型不匹配 | 数字字段输入文字、布尔字段输入数字 |
| **长度异常** | 超出范围 | 密码超短/超长、名称超长 |
| **值域异常** | 值超出有效范围 | 年龄>150、金额<0 |
| **逻辑异常** | 逻辑矛盾 | 开始日期>结束日期、库存<购买数量 |
| **重复异常** | 唯一性冲突 | 用户名已存在、订单号重复 |
| **特殊字符** | 含特殊字符 | 含SQL注入字符、含emoji、含空格 |

#### 异常场景模板

```
用例名称：<功能> - <异常类型> - <预期错误提示>
输入数据：<异常数据>
预期结果：提示具体错误信息，未执行业务逻辑
备注：<异常类型说明>
```

### 4️⃣ 错误处理 ⚠️

**定义**：系统层面的异常处理，验证系统稳定性

#### 错误处理场景表

| 错误类型 | 测试点 | 模拟方式 | 预期处理 |
|---------|-------|---------|---------|
| **权限错误** | 无操作权限 | 使用无权限账号 | 提示"权限不足"，按钮禁用 |
| **认证错误** | Token过期/无效 | 使用过期Token | 提示"请重新登录"，跳转登录页 |

### 🔄 流程场景测试模板

> 流程场景测试是验证业务流程从开始到结束是否正确执行的重要测试类型。

#### 什么是流程场景测试

**定义**：验证一个完整业务流程从前置条件开始，经过一系列操作步骤，最终达到预期结果。

**适用场景**：
- 多步骤业务流程（订单流程、审批流程）
- 状态流转场景（待支付→已支付→已发货→已完成）
- 跨模块操作（采购→入库→付款）
- 涉及多个角色的流程（申请→审批→执行）

#### 流程场景测试用例特殊字段

| 字段 | 说明 | 示例 |
|------|------|------|
| **流程名称** | 完整业务流程名称 | 订单创建→支付→发货→完成流程 |
| **流程阶段** | 当前测试阶段 | 阶段1：订单创建 |
| **前置流程** | 前序流程状态 | 订单已创建，状态为"待支付" |
| **触发条件** | 触发流程变更的条件 | 用户点击"支付"按钮 |
| **预期流转** | 预期的状态变化 | 待支付→已支付 |
| **数据变化** | 流程中数据的变化 | 库存-1，订单状态更新 |
| **通知/消息** | 流程中的通知 | 发送支付成功短信 |

#### 流程场景测试设计方法

**1️⃣ 主流程测试（Happy Path）**

验证流程在正常情况下的完整执行。

```
流程名称：订单完整流程
测试步骤：
  阶段1：创建订单
    - 选择商品，加入购物车
    - 填写收货地址
    - 提交订单
    - 验证：订单状态为"待支付"
  
  阶段2：支付订单
    - 选择支付方式
    - 完成支付
    - 验证：订单状态为"已支付"，库存扣减
  
  阶段3：商家发货
    - 点击发货，填写物流信息
    - 验证：订单状态为"已发货"
  
  阶段4：确认收货
    - 用户点击确认收货
    - 验证：订单状态为"已完成"

预期结果：
  - 状态正确流转：待支付→已支付→已发货→已完成
  - 库存正确扣减，通知正常发送
```

**2️⃣ 分支流程测试（Alternative Path）**

验证流程在不同分支情况下的执行。

| 主流程 | 分支流程 | 测试要点 |
|--------|---------|---------|
| 支付成功 | 支付失败 | 验证订单状态回滚 |
| 正常审批 | 审批驳回 | 验证流程回到申请人 |
| 普通配送 | 加急配送 | 验证费用计算差异 |
| 在线支付 | 货到付款 | 验证流程差异 |

**3️⃣ 异常流程测试（Exception Path）**

验证流程在异常情况下的处理。

| 异常类型 | 场景 | 预期处理 |
|---------|------|---------|
| 网络异常 | 支付过程中断网 | 订单状态保持一致，提示重试 |
| 超时异常 | 支付超时 | 订单自动取消或保持待支付 |
| 并发异常 | 多人同时购买同一商品 | 库存一致性保证 |
| 数据异常 | 库存不足 | 提示库存不足，无法下单 |

**4️⃣ 回退流程测试**

验证流程回退/撤销的处理。

| 场景 | 操作 | 预期结果 |
|------|------|---------|
| 取消订单 | 用户取消未支付订单 | 订单状态变为"已取消" |
| 申请退款 | 用户申请已支付订单退款 | 生成退款单，订单状态变更 |
| 撤销审批 | 审批人撤销已审批流程 | 流程回到待审批状态 |

**5️⃣ 并发流程测试**

验证多个用户同时操作同一资源的处理。

```
流程名称：并发下单库存一致性
前置条件：商品库存为1
测试步骤：
  1. 用户A和用户B同时进入商品详情页
  2. 用户A和用户B同时点击"立即购买"
  3. 用户A和用户B同时提交订单

预期结果：
  - 只有1个用户下单成功
  - 另1个用户收到"库存不足"提示
  - 库存最终为0，数据一致，无超卖
```

## 六、测试技巧与最佳实践

### 🎯 测试设计技巧

#### 1. 等价类划分

将输入数据划分为若干等价类，每个等价类选一个代表值测试。

```
示例：手机号字段（11位数字）
├── 有效等价类：11位数字 → 选 13800138000 测试
├── 无效等价类：非11位 → 选 1380013800（10位）测试
├── 无效等价类：含非数字 → 选 138abc38000 测试
└── 无效等价类：空 → 选 "" 测试
```

#### 2. 边界值分析法

测试边界值和边界附近的值。

```
示例：密码长度6-20位
测试点 = { 5位, 6位, 7位, 19位, 20位, 21位 }
         ↑最小-1 ↑最小 ↑最小+1 ↑最大-1 ↑最大 ↑最大+1
```

#### 3. 错误推测法

根据经验推测可能出错的地方。

```
常见错误点：
├── 空值处理
├── 并发操作
├── 数据类型转换
├── 时间处理（跨时区、闰年）
├── 编码问题（中文、特殊字符）
└── 边界溢出
```

#### 4. 因果图法

分析输入条件（原因）和输出结果（结果）的关系。

```
示例：登录功能
原因：账号正确(A)、密码正确(B)、账号未锁定(C)
结果：登录成功(1)、提示账号错误(2)、提示密码错误(3)、提示账号锁定(4)

组合：
A∧B∧C → 1（登录成功）
¬A → 2（账号错误）
A∧¬B → 3（密码错误）
A∧B∧¬C → 4（账号锁定）
```

#### 5. 正交试验法

减少测试组合数量，保证覆盖率。

```
示例：三个因素，每个两个水平
因素A：登录方式（账号密码/短信验证码）
因素B：网络状态（正常/弱网）
因素C：账号状态（正常/锁定）

正交组合（4次测试即可覆盖主要组合）：
1. 账号密码 + 正常网络 + 账号正常
2. 账号密码 + 弱网 + 账号锁定
3. 短信验证码 + 正常网络 + 账号锁定
4. 短信验证码 + 弱网 + 账号正常
```

#### 1. 快速识别核心场景

```
核心场景识别清单：
□ 用户最常用的功能是什么？（→ P0）
```

#### 2. 用例命名技巧

```
预期结果三段式：
1. 状态描述：成功/失败、状态码、返回值
2. 界面变化：跳转、弹窗、提示信息
3. 数据变化：字段值、数量变化

示例：
- 返回状态码200，订单ID：ORD001

- 商品库存减少1，订单状态为"待支付"
```

### 🏆 最佳实践

#### 1. 用例编写顺序

```
推荐顺序：
1. 先写 P0 正常场景（核心流程）
2. 再写必填校验（每个必填字段）
3. 再写边界场景（每个有范围的字段）
4. 再写异常场景（格式错误、逻辑错误）
5. 最后写兼容性、性能、安全
```

#### 2. 用例数量估算

```
估算公式：
基础用例数 = 功能点数 × 3（正常+异常+边界）
完整用例数 = 基础用例数 + 必填字段数 × 2 + 边界值数量

示例：登录功能
功能点：1个（登录）
必填字段：2个（账号、密码）
边界值：密码长度（6个测试点）
基础用例 = 1 × 3 = 3
完整用例 = 3 + 2 × 2 + 6 = 13条
```

#### 3. 用例评审要点

```
用例评审检查项：
□ 覆盖是否完整？（5大类场景）
□ 预期结果是否明确？（可验证）
□ 步骤是否可执行？（可复现）
□ 优先级是否合理？（P0/P1/P2/P3）
□ 是否有禁止类型用例？（需删除）
```

### 示例2：订单创建 API（完整用例集）

**API 定义**：

```
POST /api/orders
Content-Type: application/json

请求参数：
- productId (必填, string): 商品ID
- quantity (必填, integer): 购买数量，范围1-99
- address (必填, string): 收货地址，最大200字符
- couponCode (可选, string): 优惠券码

响应：
- 200: 创建成功，返回订单信息
- 400: 参数错误
- 401: 未登录
- 403: 无权限
- 404: 商品不存在
- 409: 库存不足
```

**生成的 API 测试用例**：

| API_Order_003 | quantity缺失 | POST | 无quantity | 400 |
| API_Order_004 | address缺失 | POST | 无address | 400 |
| API_Order_005 | quantity=0 | POST | quantity: 0 | 400 |
